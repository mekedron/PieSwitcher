# Release setup — from a fresh Apple Developer account to your first release

This is the **prerequisite** to running `./scripts/bootstrap-release-secrets.sh`.
The bootstrap script asks you for files and IDs you cannot get anywhere else;
this guide walks you, in order, through producing every one of them so the
bootstrap script can finish without you leaving the page.

Every artifact below is named with the **exact label the bootstrap script will
use when prompting**, so you can match a prompt to a file without translating.

You only need to do this once per machine. Re-running the bootstrap script
later re-uses everything except secrets you explicitly rotate.

---

## Before you start

You need:

- A paid Apple Developer Program membership ($99/yr). The free tier cannot
  issue a Developer ID Application certificate.
- macOS with **Keychain Access** (built-in, in `/Applications/Utilities/`).
- The [GitHub CLI](https://cli.github.com) authenticated to your GitHub
  account: `brew install gh && gh auth login`.
- Push access to the `mekedron/PieSwitcher` repository (or whichever fork
  hosts your releases) — confirm with `gh auth status`.

This guide also assumes you're working from the PieSwitcher repo root.

---

## What you'll end up with — a checklist

By the end of this guide you will have collected, in a single safe folder
(suggested: `~/PieSwitcher-release-keys/`, **excluded from cloud-sync**):

- [ ] **Developer ID Application .p12 file** — the cert+key bundle that signs the app.
- [ ] **.p12 export passphrase** — the password you set when exporting the .p12.
- [ ] **Issuer ID** — UUID of your App Store Connect API integration.
- [ ] **Key ID** — short alphanumeric ID of one specific API key.
- [ ] **.p8 file** — the App Store Connect API private key, downloadable exactly ONCE.
- [ ] **Sparkle EdDSA private key** — usually auto-generated by the bootstrap; no manual step.
- [ ] **Homebrew Tap GitHub PAT** — *optional*. Only if you want releases auto-published to your tap.

Tick each box as you finish its section, then run
`./scripts/bootstrap-release-secrets.sh` from the repo root.

> [!IMPORTANT]
> The `.p8` file (Section 2) is downloadable **exactly once**. If you miss
> the download you must revoke the key and create a new one. Read Section 2
> before clicking *Generate*.

## 1. Developer ID Application certificate (.p12)

This is the long-lived signing identity. Apple issues exactly one private key
per certificate, and that key only lives on the Mac you created it on — that
is why we export it to a `.p12` and base64 it into a GitHub secret.

### 1.1 Generate a Certificate Signing Request (CSR)

1. Open **Keychain Access** (`⌘+space` → `Keychain Access`).
2. Menu bar: **Keychain Access → Certificate Assistant → Request a Certificate
   From a Certificate Authority…**
3. Fill in:
   - **User Email Address**: the email on your Apple Developer account.
   - **Common Name**: any human-readable string (e.g., `PieSwitcher CI`).
   - **CA Email Address**: leave blank.
   - **Request is**: select **Saved to disk**.
   - Tick **Let me specify key pair information**.
4. Click **Continue**, choose a save location (e.g., `~/Desktop/CertificateSigningRequest.certSigningRequest`).
5. Key pair info dialog — leave defaults (**2048 bits**, **RSA**). Click **Continue**, then **Done**.

You now have a `.certSigningRequest` file on disk. Keychain Access has also
stored the **matching private key** in your login keychain under the name
you set as Common Name. Do not delete it — without it the certificate cannot sign.

### 1.2 Request the certificate in the Apple Developer portal

1. Visit <https://developer.apple.com/account/resources/certificates/list>.
2. Click the blue **+** (Add) button.
3. Under **Software**, choose **Developer ID Application**, click **Continue**.
4. *(If shown)* choose **G2 Sub-CA** as the profile type.
5. **Upload the CSR** you just saved. Click **Continue**.
6. Click **Download** — you get a file named `developerID_application.cer`.

### 1.3 Install the certificate into the login keychain

1. Double-click the `.cer` you just downloaded.
2. Keychain Access asks where to add it — choose **login** (default) and **Add**.
3. In Keychain Access, switch to the **login** keychain and the **My
   Certificates** category. You should see a triangle-disclosure entry named
   `Developer ID Application: <Your Name> (TEAMID)`. Expand it and confirm there
   is a private key child item underneath — that confirms the cert is paired
   with the private key Keychain generated in step 1.1.

> If the cert appears WITHOUT a private key child, the CSR was made on a
> different Mac or with a different keychain — Apple will not let you sign
> with it. Repeat steps 1.1–1.3 on the machine where the private key lives.

### 1.4 Export the cert+key as a .p12

1. In **My Certificates**, right-click the `Developer ID Application: …` row
   (the parent row that has the private key under it).
2. Choose **Export "Developer ID Application: …"…**.
3. File Format: **Personal Information Exchange (.p12)**.
4. Pick a save location — recommended: `~/PieSwitcher-release-keys/developer-id.p12`.
5. Set an **export passphrase**. Write it down — the bootstrap script will
   prompt for this exact value as **".p12 export passphrase"**. Use a strong
   random one if you also store it in a password manager.

> Save the `.p12` somewhere that is NOT synced to iCloud Drive / Dropbox /
> any cloud drive. It is a private key.

#### What it's used for

The release workflow base64-decodes this `.p12` on the CI runner, imports it
into a throw-away keychain, and uses the contained `Developer ID Application`
identity to codesign `PieSwitcher.app` (and Sparkle's nested helpers) with
the hardened runtime. Without it the released app would carry only an
ad-hoc signature and Gatekeeper would refuse to launch it.

✅ **Checklist:**
- [ ] `developer-id.p12` saved to a known path (you'll paste this path into the
      bootstrap script as **"Path to Developer ID Application .p12 file"**).
- [ ] Passphrase written down (the bootstrap will prompt for **".p12 export passphrase"** —
      input is masked).

## 2. App Store Connect API key (notarization)

Notarization uploads your signed app to Apple, who scans it for malware and
stamps a "notarization ticket" you can staple onto the .app and .dmg.
Apple authenticates the upload with an **App Store Connect API key**, NOT
your developer login. You'll generate three pieces of info:

- **Issuer ID** — one per team. Reuse across keys.
- **Key ID** — one per API key. Short, alphanumeric.
- **.p8 file** — the private key. **Downloadable exactly once.**

### 2.1 Navigate to Team Keys

1. Sign in to <https://appstoreconnect.apple.com>.
2. Top nav: **Users and Access**.
3. Sidebar: **Integrations**.
4. Top of the page: **Team Keys** (sometimes "API Keys", sometimes "App Store
   Connect API"). If you see **Individual Keys** tab open, switch to **Team Keys**
   — only Team Keys can notarize CI builds.

### 2.2 Read the Issuer ID — paste this when the bootstrap asks for "Issuer ID"

At the top of the Team Keys list there is a label **"Issuer ID"** with a UUID
next to it (e.g., `69a6de80-…`).
Click the small icon next to it to copy it; keep it on your clipboard or paste
it into a scratch note.

### 2.3 Generate the API key — gives you the Key ID and the .p8

1. Click **Generate API Key** (or the **+** button).
2. **Name**: something memorable, e.g., `PieSwitcher CI Notary`.
3. **Access**: choose **Developer** (the minimum scope that allows
   `notarytool` to submit and poll for results). Do **not** grant Admin.
4. Click **Generate**. The portal returns to the list.
5. Find your new key in the list. Note the **Key ID** column value (e.g., `ABCD1234EF`)
   — this is what the bootstrap will prompt for as **"Key ID"**.
6. The row also has a **Download API Key** link. **STOP** and read 2.4 before clicking.

> [!WARNING]
> **The "Download API Key" link works exactly ONCE.** Once you click it the
> download starts and Apple removes the file. If you lose it you cannot
> re-download it — you must revoke the key and create a new one.

### 2.4 Download the .p8 — handle with care

1. Click **Download API Key**. You get a file named like
   `AuthKey_ABCD1234EF.p8`.
2. Move it **immediately** to a safe location, e.g.,
   `~/PieSwitcher-release-keys/AuthKey_ABCD1234EF.p8`.
3. Set restrictive permissions: `chmod 600 ~/PieSwitcher-release-keys/AuthKey_*.p8`.
4. Back it up to a password manager that supports file attachments (1Password
   document, Bitwarden file attachment, etc.). Do not upload to plain cloud
   storage.

#### What it's used for

The release workflow base64-decodes this `.p8` on the CI runner and passes
it to `xcrun notarytool submit … --key AuthKey.p8 --key-id … --issuer …` to
notarize the signed `.app` and `.dmg`. Without it the release can still
build and sign, but it cannot be stapled — Gatekeeper would do an online
check at first launch and might still warn.

✅ **Checklist:**
- [ ] Issuer ID saved (bootstrap prompt: **"Issuer ID"**).
- [ ] Key ID saved (bootstrap prompt: **"Key ID"**).
- [ ] `AuthKey_<KEY_ID>.p8` saved to a safe path (bootstrap prompt: **"Path to .p8 file"**).

## 3. Sparkle EdDSA keypair

Sparkle (the in-app updater) verifies update downloads with an Ed25519
signature. The keypair has two parts:

- A **private key** the release workflow signs DMGs with.
- A **public key** embedded in `PieSwitcher/Info.plist` as `SUPublicEDKey`,
  which every shipped app uses to verify update signatures.

### What the bootstrap does for you

If `sparkle_private_key` (in the repo root) is missing or empty, the
bootstrap script **auto-generates a fresh keypair**, writes the private key
to `sparkle_private_key` (which is gitignored), uploads it as the
`SPARKLE_PRIVATE_KEY` secret, and prints the matching public key so you can
paste it into `PieSwitcher/Info.plist` as `SUPublicEDKey`.

If `sparkle_private_key` already exists, the script re-uses it — that is the
expected case for the PieSwitcher repo today, which already ships the key
file and a matching `SUPublicEDKey` in Info.plist.

### What you must do manually

In the normal case: **nothing.** Just make sure that after the bootstrap
runs:

1. `sparkle_private_key` exists in the repo root and is **not** committed
   (it's covered by `.gitignore`).
2. The `SUPublicEDKey` value in `PieSwitcher/Info.plist` matches the
   private key (the bootstrap prints a warning if it spots a mismatch).
3. **Back up `sparkle_private_key`** to a password manager. If you lose it,
   you cannot publish updates that existing installs will accept — they'll
   reject any new DMG signed with a fresh key, because they only trust the
   old public key embedded in the app they were shipped with. Rotating the
   key requires shipping a new app build with a new `SUPublicEDKey`.

#### What it's used for

`SPARKLE_PRIVATE_KEY` is used by `sparkle-dist/bin/sign_update` in the
release workflow to produce the `sparkle:edSignature` attribute in
`appcast.xml`. Sparkle running inside installed apps verifies that signature
against the embedded `SUPublicEDKey` before installing an update.

✅ **Checklist:**
- [ ] `sparkle_private_key` exists in repo root (bootstrap will create if missing).
- [ ] `SUPublicEDKey` in `PieSwitcher/Info.plist` matches it (bootstrap warns if not).
- [ ] Backed up the private key file to a password manager.

## 4. Homebrew Tap GitHub PAT (optional)

The release workflow can auto-update a `pieswitcher` cask in a separate
Homebrew tap repository (e.g., `mekedron/homebrew-tap`). This is **opt-in**.
If you don't have a tap yet, run the bootstrap with `--skip-homebrew` (or
answer "no" at the prompt) — every other step still works and the workflow's
Homebrew step becomes a no-op.

### If you want Homebrew distribution

1. Create the tap repo if it doesn't exist yet — typically
   `github.com/<your-user>/homebrew-tap`. Add a `Casks/` directory; the
   workflow writes `Casks/pieswitcher.rb` automatically on each release.
2. Visit <https://github.com/settings/tokens?type=beta> (fine-grained PATs).
3. Click **Generate new token**.
4. **Token name**: `pieswitcher-homebrew-release`.
5. **Expiration**: pick the longest acceptable horizon (Apple-style: 1 year is
   a sensible default).
6. **Resource owner**: the account/org that owns the tap repo.
7. **Repository access**: **Only select repositories** → pick the tap repo.
8. **Permissions** → **Repository permissions**:
   - **Contents**: **Read and write** (needed to commit cask updates).
   - Everything else: leave default (no access).
9. Click **Generate token** and copy the token string.
10. Paste it into the bootstrap when prompted for the **"Homebrew Tap GitHub PAT"**
    (input is masked).

> Classic PATs with `repo` scope work too if your tap is on an account that
> doesn't support fine-grained PATs (e.g., some legacy orgs). Treat the token
> like a password.

#### What it's used for

`HOMEBREW_TAP_TOKEN` is used by the **Update Homebrew Cask** step in the
release workflow to PUT a new `Casks/pieswitcher.rb` into your tap repo via
the GitHub Contents API on each tag push. Without it, that step is skipped
(detected via `secrets.HOMEBREW_TAP_TOKEN != ''`).

✅ **Checklist:**
- [ ] Tap repo exists (or you're skipping).
- [ ] PAT generated (bootstrap prompt: **"Homebrew Tap GitHub PAT"**).
      OR: planning to pass `--skip-homebrew`.

## Final checklist — run this before the bootstrap

You should have, in `~/PieSwitcher-release-keys/`:

- [ ] `developer-id.p12`
- [ ] `developer-id.p12` passphrase (in your password manager)
- [ ] `AuthKey_<KEYID>.p8`
- [ ] Issuer ID (in your password manager / scratch note)
- [ ] Key ID (in your password manager / scratch note)
- [ ] *(optional)* Homebrew Tap PAT (in your password manager)

And in the repo:

- [ ] `sparkle_private_key` exists OR you're happy for the bootstrap to make one.
- [ ] `PieSwitcher/Info.plist` already contains `SUPublicEDKey` (it does in
      this repo — the bootstrap only complains if you regenerate the keypair).

Now run, from the repo root:

```sh
./scripts/bootstrap-release-secrets.sh
# or, if you don't want a Homebrew tap yet:
./scripts/bootstrap-release-secrets.sh --skip-homebrew
```

When it finishes, push a tag matching `v*.*.*` (e.g., `git tag v0.1.0 && git
push origin v0.1.0`) to trigger the release workflow.
