## Codebase Patterns (Study These First)

### project.pbxproj is hand-written with sequential hex IDs
- Object IDs follow `1A00000000000000000000XX` where `XX` is hex, assigned in order. Highest used so far: `0x20`. Continue the sequence for new objects.
- Adding a **source file** to a target requires editing 4 spots: `PBXFileReference`, `PBXBuildFile`, the owning `PBXGroup`'s `children`, and the target's `PBXSourcesBuildPhase` `files`. There is no Xcode GUI doing this for you here.
- After editing, sanity-check with `xcodebuild -list -project Bringr.xcodeproj` (parses the project; lists targets/schemes).

### Unit testing setup
- Test target `BringrTests` is **host-based**: `TEST_HOST = $(BUILT_PRODUCTS_DIR)/Bringr.app/Contents/MacOS/Bringr`, `BUNDLE_LOADER = $(TEST_HOST)`, plus a `PBXTargetDependency` on the app. `ENABLE_TESTABILITY = YES` is set at project Debug level, so `@testable import Bringr` works.
- To make an app type reachable from tests, keep it `internal` (the default). `private` members are NOT visible even via `@testable` — e.g. `AboutView()` is uncallable from tests because a private stored property makes its memberwise init private.
- The test target is wired into the shared scheme's `<Testables>` (else `xcodebuild test` finds nothing to run).
- **GOTCHA for US-007/US-008:** `xcodebuild test` launches the host app (Bringr.app) to inject the bundle. Once app launch installs a global `CGEventTap` / MultitouchSupport observer, guard it so it does NOT run under XCTest (check env var `XCTestConfigurationFilePath` or `NSClassFromString("XCTest")`), or tests may hang or trigger Accessibility/Input-Monitoring permission prompts.

### Quality gates (run all three before "done")
- Build: `xcodebuild -project Bringr.xcodeproj -scheme Bringr -configuration Debug -derivedDataPath build build`
- Lint: `swiftlint lint --strict` (binary from `brew install swiftlint`; config `.swiftlint.yml` uses `included: [Bringr, BringrTests]` so it never scans `build/`). `--strict` turns warnings into failures.
- Test: `xcodebuild test -project Bringr.xcodeproj -scheme Bringr -destination 'platform=macOS'`

### App lifecycle, @MainActor services, and launch bootstrap
- `AppDelegate` (`Bringr/AppDelegate.swift`) is the launch-time bootstrap home, wired via `@NSApplicationDelegateAdaptor(AppDelegate.self)` in `BringrApp`. It owns app-lifetime services and is where future overlay pre-warm (US-006) and event taps (US-007/008) belong. There is no clean pure-SwiftUI "on launch once" hook for a MenuBarExtra-only app.
- **Concurrency gotcha:** a `@MainActor` `ObservableObject` service cannot be a default-initialized stored property of a *non*-isolated `NSObject` (its init is main-actor isolated). Fix: mark the owning `AppDelegate` `@MainActor` too. App delegates run on the main actor anyway.
- **Concurrency gotcha:** static helpers used as **default closure arguments** (`init(probe: () -> Bool = Self.systemFoo)`) must be `nonisolated`, else "Converting `@MainActor () -> X` ... loses global actor" (warning in Swift 5, error in Swift 6).
- **Testability seam:** wrap live system calls (`AXIsProcessTrusted`, prompts, `NSWorkspace.open`) in injectable closures with live-impl defaults. Tests construct the service with fixture closures — no real permission state, no system dialogs. See `PermissionsManager`.
- **XCTest launch guard (the US-001 gotcha, now concrete):** `AppDelegate.applicationDidFinishLaunching` early-returns when `ProcessInfo.processInfo.environment["XCTestConfigurationFilePath"] != nil`. `xcodebuild test` launches Bringr.app to host the bundle; without this guard the launch-time AX prompt pops a system dialog / hangs the run. Reuse this exact guard for the event tap (US-007) and multitouch observer (US-008).
- **Sharing a service across AppDelegate + SwiftUI scenes:** the delegate creates the instance; scenes read it via `appDelegate.permissions` and inject with `.environmentObject(...)` (or pass to a `@ObservedObject` init param, as `MenuContent` does). One instance, observed live.

### Launch-time AppKit modal in an LSUIElement app (the US-002 hardening / re-grant pattern)
- For a floating window that must come to front from a menu-bar-only app (e.g. the permission re-grant alert), an **AppKit `NSWindow` subclass** beats a SwiftUI `Window` scene: you get `level = .floating`, `isReleasedWhenClosed = false`, transparent titlebar, and a `moveAside()` (`setFrame(_:display:animate:)` to `screen.visibleFrame.minX + 20`) so it doesn't cover the system prompt. Style mask `[.titled, .closable, .fullSizeContentView]` + `titlebarAppearsTransparent`/`titleVisibility = .hidden`.
- **Ownership lives on `AppDelegate`** (mirrors ClipSlop's AppState): the delegate holds `var window: PermissionAlertWindow?`, builds it lazily, `center()` + `DispatchQueue.main.async { makeKeyAndOrderFront(nil); NSApp.activate(ignoringOtherApps: true) }`, and a `dismiss()` that `close()`es + nils the ref. `close()` on an already-closed window (ref nil) is safe, so double-dismiss (e.g. auto-dismiss racing a manual Close) is fine.
- **Host SwiftUI via `NSHostingView`** and pass `() -> Void` callbacks for the window-only bits (`onMoveAside`, `onClose`); call the `@MainActor` service (`permissions.requestAccess()`/`.openAccessibilitySettings()`) **directly in the view** — same proven pattern as `PreferencesView`'s `permissions.recheck()`, no actor-isolation warnings. Capturing `[weak self]` in the closures formed inside the window's `@MainActor init` is MainActor-isolated (non-`@Sendable`), so calling `self?.moveAside()` compiles clean.
- **Auto-dismiss by observing `@Published`**: `@ObservedObject var permissions` + `.onChange(of: permissions.isTrusted) { _, trusted in if trusted { onClose() } }` closes the window the instant `PermissionsManager.startMonitoring()` (1s poll + `com.apple.accessibility.api` notification) flips trust — no manual "validate" button, no relaunch. The window is only ever constructed while untrusted (guarded by a pure `nonisolated static shouldPresentPermissionAlert(isTrusted:suppressed:)`), so the change always fires.
- **First persisted setting**: `@AppStorage("suppressPermissionAlert")` in the view; `AppDelegate` reads the same key via `UserDefaults.standard.bool(forKey:)`. Keep the key in ONE place — `nonisolated static let suppressDefaultsKey` on the window class — and reference it from both the view's `@AppStorage(...)` and the delegate, so they can't drift (covered by a test).

### Window/app control behind an injectable seam (the US-004 pattern)
- The PermissionsManager injectable-closure pattern generalizes to a **protocol seam** for anything that touches live system state. `WindowControlling` (a `@MainActor protocol`) is the seam: `WindowController` (orchestration) is unit-tested against an in-memory `FakeWindowSystem`; `LiveWindowSystem` (AX + NSRunningApplication) is the production conformer. Tests never construct `LiveWindowSystem`, so no AX calls / permission prompts during `xcodebuild test`.
- **Identity types** for windows/apps: `AppID { pid }` and `WindowID { app; token }`, both `Hashable, Sendable`. `token` is opaque (live system uses enumeration index for now). US-003's enumeration service can refine `token` later — `WindowController` only relies on `Hashable`, so it's additive. Don't over-commit US-004 to an ID scheme US-003 owns.
- **Capture-before-mutate / restore-to-baseline**: capture the pre-mutation state of the touched scope (app visibility+frontmost, or one app's window minimized-state+order) **exactly once per session** (guard flags: `didCaptureApps`, `windowBaseline[app] == nil`). `restore()` replays the captured baseline, NOT the inverse of each mutation — this is why re-targeting (hover app A → app B → A) restores correctly: incremental "undo what I did" breaks under re-targeting, baseline-replay doesn't. Restore order: unhide apps → un-minimize + re-raise windows (back-to-front so original front ends on top) → activate original frontmost last.
- **Hiding one window**: AX has no "hide window"; `kAXMinimizedAttribute = true` is the only reversible per-window hide. `restore()` un-minimizes. (OS minimize animation is system behavior, unrelated to the PRD's "no wheel animations".)
- **Live AX boilerplate that works under strict build** (no bridging header, no private API): app element via `AXUIElementCreateApplication(pid)`; windows via `AXUIElementCopyAttributeValue(_, kAXWindowsAttribute as CFString, &v)` then `v as? [AXUIElement]`; read bool attrs with `value as? Bool`; set bools with `kCFBooleanTrue/kCFBooleanFalse` (NOT `true as CFBoolean`); raise via `AXUIElementPerformAction(el, kAXRaiseAction as CFString)`; focus via setting `kAXMainAttribute` + `kAXFocusedAttribute`. kAX* constants import as `String`. AX symbols link with just `import ApplicationServices` (no explicit framework).
- **Same @MainActor-default-arg gotcha as before, non-closure form**: `init(system: WindowControlling = LiveWindowSystem())` fails ("Call to main actor-isolated initializer in a synchronous nonisolated context"). Fix: `init(system: WindowControlling? = nil) { self.system = system ?? LiveWindowSystem() }` — construct inside the `@MainActor` init body.
- **Verification limitation**: US-004 ships primitives with no runtime driver yet (no trigger/overlay until US-006/007/010), so "build & run verify visually" only confirms the app still launches stably with the new code linked. The live AX primitives get interactive verification when a trigger exists; orchestration is proven by the fake-backed tests now.

### Window enumeration behind a source seam (the US-003 pattern)
- Same seam shape as `WindowControlling`: `WindowEnumerationSource` (`@MainActor protocol`, exposes `rawWindows() -> [RawWindow]` + `selfPID`) is the seam. `WindowEnumerator` holds ALL grouping/filtering logic and is unit-tested against an in-memory `FakeWindowEnumerationSource`; `CGWindowSource` (live, CoreGraphics) is the production conformer. Tests never construct `CGWindowSource`. Same `init(source: WindowEnumerationSource? = nil) { source ?? CGWindowSource() }` trick to dodge the `@MainActor` default-arg error.
- **Keep the live layer dumb.** `CGWindowSource` just maps `CGWindowListCopyWindowInfo([.optionOnScreenOnly, .excludeDesktopElements], kCGNullWindowID)` dicts → flat `[RawWindow]` value structs. No grouping, no filtering there. That puts maximum logic (exclude self/non-normal, group-by-pid preserving front-to-back order, title fallback) in the testable `WindowEnumerator`.
- **CGWindowList dict access:** read numbers via `(info[kCGWindowNumber as String] as? NSNumber)?.intValue` / `.int32Value` (NOT `as? Int`/`as? pid_t` — bridging is finicky). Bounds: `info[kCGWindowBounds as String] as? NSDictionary` then `CGRect(dictionaryRepresentation: dict as CFDictionary)`. CG constants are `CFString`; use `as String` as the dict key.
- **"Normal window" filter (heuristic):** `layer == 0` (drops menu bar/dock/status items/panels), `alpha > 0` (drops invisible helpers), non-empty `ownerName`, and width/height ≥ `minimumWindowSize` (40). Apps left with zero surviving windows drop out → satisfies "exclude menu-bar-only apps/agents" for free (no NSWorkspace activationPolicy needed; CG already returns only apps with on-screen windows).
- **Stable id = `kCGWindowNumber`** → `WindowID(app:, token: windowNumber)`, stable for the window's lifetime (real improvement over US-004's enumeration-index token). **⚠️ INTEGRATION SEAM for US-010/011/012:** `LiveWindowSystem.windows(of:)` (US-004) still mints `WindowID(token: enumerationIndex)` and caches AX elements by it — a DIFFERENT scheme. A `WindowID` from `WindowEnumerator` will NOT be found in `LiveWindowSystem`'s cache. When wiring enumerator output into `WindowController`, reconcile them (e.g. have `LiveWindowSystem` resolve its AX element by matching CG window number, or drive both from one source). Don't assume the two `WindowID`s are interchangeable yet.
- **Titles are best-effort in v1:** `kCGWindowName` needs Screen Recording permission (a non-goal), so titles are usually empty under Accessibility-only; `WindowEnumerator.title(for:index:)` falls back to `"Window N"` (US-006 permits index labels). Real titles are a clean later enrichment: fill `RawWindow.title` from AX in the live layer — model/consumers don't change.
- **Timing (AC4):** `WindowEnumerator.lastDuration` (+ os_log) records each `enumerate()` call. Measured live ~0.7 ms for 8 apps (CoreGraphics path), far under the 16 ms summon budget. Spaces caveat: `.optionOnScreenOnly` returns only the current Space's windows — apps with windows only on other Spaces won't appear (acceptable for v1 "live state at summon"; ties to the PRD multi-monitor/Spaces open question).

### Radial overlay + pure layout geometry (the US-006 pattern)
- **Pure value-type geometry core holds ALL the math; window/view/controller are thin shells.** `RadialLayout.swift` (`RadialGeometry`, `RadialLayout`, `RadialMenuPlacement`) is CoreGraphics-only (no AppKit/SwiftUI) and fully unit-tested; appearance is verified by build & run. Same split philosophy as the `WindowControlling`/`WindowEnumerationSource` seams.
- **Layout convention:** slice 0 centred at top (12 o'clock), advancing clockwise. Offsets + the hit-test point are measured from the ring centre in **y-down** space (+x right, +y down) so they map onto SwiftUI drawing coords with no flip. Angle clockwise-from-north = `atan2(offset.x, -offset.y)`. Slice index = `Int((normalizedAngle + halfSpan) / sliceSpan) % itemCount` — the `+halfSpan` centres slice 0 on north, the `% itemCount` wraps the seam at north (so a point a hair CCW of north is still slice 0, not out of bounds). `hitTest` returns `nil` for both dead-zone (`< innerRadius`) and outside (`> outerRadius`) — a single "none".
- **Placement coordinate gotcha:** `NSEvent.mouseLocation` and `NSWindow.setFrameOrigin` are BOTH AppKit global y-up, so `windowOrigin = (cursor.x - w/2, cursor.y - h/2)` centres with no flip, and centring on the cursor inherently lands the wheel on the display under the cursor. BUT **CGWindowList reports bounds in CG top-left (y-down)**: when verifying, x matches your origin exactly (confirms centring) but y is flipped (`CG_y = totalHeight - appkit_y - height`) — don't expect y to match.
- **Pre-warmed overlay = `NSPanel`, not `NSWindow`.** `[.borderless, .nonactivatingPanel]`, `isOpaque=false`, `backgroundColor=.clear`, `hasShadow=false`, `level=.floating`, `isFloatingPanel=true`, `hidesOnDeactivate=false`, `isReleasedWhenClosed=false`, `collectionBehavior=[.canJoinAllSpaces,.stationary,.fullScreenAuxiliary]`. **Nonactivating is the key choice**: summoning over Chrome must NOT activate Bringr / deactivate Chrome (that would disturb the very window state the switcher must preserve, US-004/012/015). Build `NSHostingView(rootView:)` ONCE in the controller init (pre-warm); summon only swaps `@Published slices`, repositions, `orderFrontRegardless()` — no allocation on the hot path (FR-14). Dismiss = `orderOut(nil)` (shown/hidden, never recreated).
- **Controller↔window↔view retain cycle is intentional & fine**: `RadialMenuController` (`@MainActor ObservableObject`) → window → hosting view → (`@ObservedObject`) controller. It's an app-lifetime object owned by `AppDelegate` forever, same model as the delegate holding `permissions` — not a growing leak.
- **Wedge rendering dodges SwiftUI's arc-angle convention**: build the wedge `Path` by sampling `layout.ringPoint(angle:radius:)` (the SAME tested math) along the outer arc then back along the inner arc, rather than `Path.addArc` (whose `clockwise:` flips under y-down). Keeps render in lock-step with hit-testing, no convention reasoning needed. (Since screenshots fail headlessly, exact wedge pixels are unverifiable anyway — sampling the trusted math is the safest way to be correct-by-construction.)
- **First real runtime wiring of US-003 + US-005**: `AppDelegate.prewarmRadialMenu()` builds `WindowEnumerator` → `WindowSwitcherMenu`, registers it under BOTH triggers in a `MenuRegistry`, hands that to `RadialMenuController`. A menu-bar "Open Window Switcher" item calls `controller.toggle(trigger:.mouseChord, at: NSEvent.mouseLocation)` as the v1 entry point / fallback until the global triggers (US-007/008) land. Pre-warm sits inside the SAME `isRunningTests` early-return as the permission bootstrap, so no NSPanel is created during `xcodebuild test` (`radialMenu` stays `nil`; `MenuContent` takes it as an optional and just hides the item).
- **Headless verification of the *visible* overlay**: it only shows on summon (a menu click you can't do headlessly). Temporarily gate an auto-summon on an env var (`BRINGR_DEBUG_SUMMON`) in `applicationDidFinishLaunching`, launch with it set, then CGWindowList-confirm a 320×320 (=2·outerRadius; borderless ⇒ NO titlebar chrome, vs the permission modal's +~32px) layer-3 window at the expected x — then REVERT the hook. Same throwaway-and-delete discipline as the US-002/016 CGWindowList script.

### Global mouse-chord activation: pure detector + live CGEventTap (the US-007 pattern)
- **Same seam split as before**: a pure value-type `MouseChordDetector` (struct, `mutating func handle(_:)`/`handleTimeout(at:)`/`reset()`) holds ALL the threshold/combo state machine and is 100% unit-tested with synthetic `MouseButtonEvent`s — zero `CGEvent`/live dependency (AC5). The live `MouseChordMonitor` (`@MainActor`) installs the `CGEventTap`, maps `CGEventType` → value events, and applies the detector's `MouseChordReaction`. Both live in one file `MouseChordActivation.swift` (mirrors `WindowControl.swift` holding protocol+orchestrator+live conformer together → fewer pbxproj edits).
- **AC4 "don't leak the chord" forces deferral.** You CANNOT decide on the first button-down whether it's a lone click or a chord-start, and a leaked right-down pops a context menu behind the overlay. So the detector returns `.hold` for the first press (live layer buffers the `CGEvent`, swallows it for now). The hold resolves the instant the answer is known: partner-down within threshold → `.summon` (discard the buffered first → it NEVER reached the app, no leak; consume the 2nd down + both ups); same-button up → `.releaseHeldWithCurrent` (append the up to the buffer, replay `[down,up]` in order, drop original → clean normal click); threshold timer fires → `handleTimeout` true → replay the lone down (press-and-hold/drag). Late partner (> threshold) → `.releaseHeldThenHold` (replay old, hold new). This is the ONLY model that satisfies AC2 (normal clicks pass) AND AC4 (no leak) together.
- **Latency trade-off (documented, tunable via threshold):** quick taps resolve at their own up (replay on up beats the timer), so ~their click duration; press-and-hold incurs up-to-`threshold` lag before the down reaches the app. Default threshold 0.12s. Replaying on up (not waiting for the timer) is what keeps tap latency low.
- **CGEventTap + Swift concurrency:** the `CGEventTapCallBack` is `@convention(c)` so it CANNOT capture — pass `self` via `userInfo: Unmanaged.passUnretained(self).toOpaque()`, recover with `Unmanaged<Monitor>.fromOpaque(_).takeUnretainedValue()`. The tap source is added to `CFRunLoopGetMain()`, so the callback fires on the main thread → wrap the body in `MainActor.assumeIsolated { ... }` (returns the `Unmanaged<CGEvent>?`) to call `@MainActor` methods cleanly. Re-enable on `.tapDisabledByTimeout`/`.tapDisabledByUserInput`.
- **Replay without infinite re-detection:** stamp re-injected events with `setIntegerValueField(.eventSourceUserData, value: sentinel)` before `event.post(tap: .cgSessionEventTap)`; at the top of the callback, if `getIntegerValueField(.eventSourceUserData) == sentinel` return `passUnretained(event)` (let our own replays through, don't feed the detector). Return `nil` to consume, `Unmanaged.passUnretained(event)` to pass.
- **Tap needs Accessibility; degrade + retry like US-002.** `CGEvent.tapCreate` returns `nil` when untrusted → `start()` returns `false`, logs, no crash. AppDelegate retries via `permissions.$isTrusted.sink { if $0 { monitor.start() } }` (needs `import Combine` + an `AnyCancellable`), so granting access re-installs with no relaunch. `start()`/`stop()` are idempotent. Guarded under the SAME `isRunningTests` early-return as the rest of the launch bootstrap (no tap during `xcodebuild test`).
- **Wiring:** `onChord` calls `radialMenu.toggle(trigger: .mouseChord, at: NSEvent.mouseLocation)` (AppKit y-up coords, matches `RadialMenuPlacement`); toggle (not summon) so a second chord also dismisses, parity with the menu-bar item.
- **Headless verification of the live tap:** os_log/`Logger` is FULLY suppressed in this session — `log show`/`log stream` (even `--info --debug`) return nothing, and Logger.info isn't persisted. To confirm a launch-time code path actually ran, temporarily add an `NSLog("BRINGR-DEBUG ...")` (NSLog hits stderr when you run the binary DIRECTLY, not via `open`), run `build/.../Bringr.app/Contents/MacOS/Bringr 2>&1 | grep BRINGR-DEBUG`, then REVERT. Confirmed `tap started = false` on the untrusted dev build (graceful degrade works) + app stays alive. The trusted-path chord/suppression + overlay can't be exercised headlessly (no GUI, can't grant TCC non-interactively); detection logic is covered by the unit tests instead.

### Interaction modes as a pure state machine (the US-009 pattern)
- **Same pure-core/thin-shell split as the detector:** all open/select/cancel policy lives in a value-type `InteractionStateMachine` in `InteractionMode.swift` — 100% unit-tested with synthetic `InteractionInput`s (`.triggerPressed`, `.triggerReleased(over:)`, `.click(over:)`, `.escape`) returning `InteractionOutcome` (`.none`/`.open`/`.select(Int)`/`.cancel`). `RadialMenuController` is a thin shell that only translates live triggers/clicks into inputs and performs the outcome's side effect. No AppKit in the machine → AC5 "state machine unit-tested" is trivial.
- **Both modes funnel through ONE `commit(for: SliceTarget)` helper** (`.slice(i)`→`.select(i)`, `.none`→`.cancel`). The ONLY difference between modes: which input is the commit gesture — hold-to-select commits on `.triggerReleased`, click-to-stay commits on `.click` (and ignores the release, staying open). That is exactly AC4's "same selection/cancel paths, differ only in open/persist semantics." Re-trigger while open → `.cancel` (toggle-dismiss parity with the menu-bar item).
- **`SliceTarget` is "slice index or none"** — `.none` covers BOTH the central dead zone and outside-the-ring (the layout's `hitTest` already returns one `nil` for both). So "release in dead zone cancels" and "click off any slice cancels" are the same code path.
- **Mode persistence mirrors `suppressDefaultsKey`:** `InteractionMode` (`String`-raw, `CaseIterable`) owns `static let defaultsKey = "interactionMode"` (single source of truth) + `static func current(from: UserDefaults = .standard)` (reads, falls back to `.default = .holdToSelect`). Preferences binds a `.radioGroup` `Picker` via `@AppStorage(InteractionMode.defaultsKey)`; the controller reads `modeProvider: () -> InteractionMode = { InteractionMode.current() }` AT EACH summon → AC3 "takes effect on next summon, no relaunch." `current` is a plain `static` (nonisolated) so it's safe as a default closure arg on the `@MainActor` controller init (the recurring default-arg gotcha).
- **Mode is snapshotted at open, not mid-session:** the controller sets `machine.mode = modeProvider()` only `if !machine.isOpen` before feeding `.triggerPressed`. The machine itself has a public `var mode`; the "next summon" guarantee is the controller's guard, not the machine's (so the machine tests just set `mode` directly to drive each mode).
- **Hold-mode needs the chord's RELEASE, which US-007's monitor didn't emit.** Added `onChordReleased` to `MouseChordMonitor` WITHOUT touching the pure `MouseChordDetector` or its 15 tests: the monitor tracks its own `physicallyDown: Set<MouseButton>` from the raw event stream (sentinel replays skipped, so they don't count) and a `chordActive` flag set on `.summon`; fires `onChordReleased` once when `chordActive && physicallyDown.isEmpty`. `AppDelegate` wires `onChord`→`triggerPressed(for:.mouseChord,at:)`, `onChordReleased`→`triggerReleased(at:)`.
- **Cursor→slice in two coordinate spaces:** chord-release uses the GLOBAL cursor (`NSEvent.mouseLocation`, AppKit y-up) → flip to layout y-down via `(cursor.x - window.midX, window.midY - cursor.y)`; the overlay click uses a LOCAL SwiftUI point (already y-down, top-left origin) → `(local.x - d/2, local.y - d/2)`. Both feed the same `layout.hitTest`. The overlay click comes from a `DragGesture(minimumDistance: 0).onEnded { $0.location }` (replaces the old `.onTapGesture`, which gave no location).
- **Scope boundary with US-015:** US-009 ships the machine that *handles* `.escape` and `.click(.none)` (tested), and the in-window cancels (hold release in dead-zone; click the wheel's dead-zone). The LIVE Esc keypress + click-OUTSIDE-the-window monitors are US-015's explicit charter ("Esc, click-outside, release-in-dead-zone, trigger-loss all cancel cleanly") — deliberately NOT wired here to avoid unverifiable, soon-reworked global monitors. `commitSelection(at:)`/`cancelInteraction()` are distinct seams (dismiss now; US-012 focus+remember / US-015 restore later).
- **Default = `.holdToSelect`** (documented in the enum): most fluid match for the hold-natured mouse chord. Menu-bar "Open Window Switcher" is a click with no hold, so it calls `summonFromMenuBar` which FORCES click-to-stay regardless of the setting (otherwise hold-mode-via-menu-bar would open a wheel with no release to commit and clicks ignored = stuck).

### Concentric hover navigation (the US-010 pattern)
- **Same pure-core/thin-shell split, third time:** ALL drill-in/re-target/collapse policy + concentric geometry lives in a value-logic `RadialNavigator` (`@MainActor`, NO AppKit window) in `RadialNavigator.swift`, unit-tested against `FakeWindowSystem` + `StubEnumerationSource`. `RadialMenuController` is the thin shell (window, hosting view, global mouse monitor) that feeds it cursor regions and mirrors its `rings`/`hovered` into its own `@Published` so the view still binds ONE `ObservableObject`. The controller is still NOT constructed in tests (it builds an NSPanel); the navigator is the testable seam.
- **The sub-wheel is a CONCENTRIC OUTER ring, not a replace-the-wheel drill.** AC3 ("moving to a different app re-targets") REQUIRES the apps ring to stay hoverable while the windows sub-wheel is shown, so apps live on ring 0 ([55,160]) and the hovered app's windows on ring 1 ([160,265]) — the cursor's *distance* picks the level, its *angle* picks the slice. Replace-the-wheel can't satisfy AC3 (you'd have to back out first).
- **Bands must TOUCH (gap = 0).** ring(level) = `[baseInner + level·thickness, +thickness]`, thickness = base `(outer-inner)`. A gap between rings is a `.none` band you fall through while gliding outward from an app to its windows → it would collapse the sub-wheel mid-gesture. Touching bands make the glide continuous; at the exact boundary the inner ring wins (hit-test iterates innermost-first). `overallDiameter = 2·outerRadius(maxDepth-1)` (530 for 2 levels); the panel is pre-warmed at that fixed size and only the active rings are drawn, so summon never resizes (FR-14 intact). `RadialLayout`/`RadialGeometry` stay single-ring & untouched — bands are derived in the navigator, so RadialLayoutTests don't change.
- **Multi-ring hit-test = iterate rings innermost-first, first `RadialLayout(geometry: ring.band).hitTest(offset)` non-nil wins** → `HoverRegion.slice(level:index:)`; none matches → `.none`. One offset→region function serves BOTH live hover and the commit-target mapping.
- **Hover needs a GLOBAL monitor, not SwiftUI `.onContinuousHover`.** The overlay is a non-activating panel, so moves go to the app underneath; and in hold-to-select the chord buttons are physically held, so moves may arrive as `*MouseDragged`. `NSEvent.addGlobalMonitorForEvents(matching: [.mouseMoved,.leftMouseDragged,.rightMouseDragged,.otherMouseDragged]) { updateHover(forGlobalCursor: NSEvent.mouseLocation) }` (started on summon, `removeMonitor` on dismiss) covers both modes and both button states. Global mouse monitors don't need Accessibility. (This is US-010's own hover wiring — distinct from the live Esc/click-OUTSIDE cancel monitors still owed by US-015.)
- **Re-target is just calling `hideOtherApps(besides: newApp)` again** — `WindowController`'s capture-once baseline (US-004) makes A→B unhide B + hide A with no restore in between (no flicker); only a move to the dead zone calls `restore()` (ends the session; a later re-expand captures a fresh baseline = the now-restored state). So baseline-replay survives arbitrary hover churn.
- **App-level reveal is pid-only (`AppID`), dodging the US-003↔US-004 `WindowID` token mismatch** and needing no Accessibility (`NSRunningApplication.hide/unhide/activate`). US-011/012 (window-level) still must reconcile the two WindowID schemes. "Apply the chosen reveal strategy" (US-013) ships here as hide-others only; US-013 generalizes.
- **AC5 "no two-level hack":** the sub-ring is `appNode.resolvedChildren()` (the US-005 dynamic provider, re-resolved on every (re-)target → live windows), not a bespoke window-list. v1 is 2 levels deep because the v1 tree is; the mechanism (resolvedChildren → next concentric ring) generalizes. Windows are leaves (`.focusWindow`, no children), so only the apps ring auto-expands in v1.

### Window-level isolation + the US-003↔US-004 WindowID reconciliation (the US-011 pattern)
- **US-011 is the story that finally reconciles the two `WindowID` token schemes** (flagged since US-003). The fix: make BOTH subsystems use `kCGWindowNumber` as the token. The enumerator already did; `LiveWindowSystem.windows(of:)` now derives each AX window's CG number via the Accessibility SPI `_AXUIElementGetWindow` (declared `@_silgen_name("_AXUIElementGetWindow")` — links against the already-imported Accessibility framework, **builds clean, swiftlint-clean** because the Swift identifier is `axUIElementGetWindow` and the underscore name is just the attribute's string literal). Falls back to the enumeration index if the SPI fails (degrade, never crash). Now a `WindowID` from a `.focusWindow` menu node resolves to the right AX element in `LiveWindowSystem`'s cache — the menu, enumerator, and controller all agree. **No heuristic bounds/title matching needed; `_AXUIElementGetWindow` is the standard, reliable AX→CGWindowID map every macOS window manager uses.**
- **Window-hover lives in the SAME `RadialNavigator` core as app-hover** — just a new `updateHover` case for `.slice(level: 1, index:)` → `isolateWindow(at:)` → `windowControl.hideOtherWindows(besides:)` (the US-004 primitive). The target `WindowID` is pulled straight out of the hovered node's action: `guard case .focusWindow(let id) = node.action`. Re-isolation between window slices is FREE: `hideOtherWindows`'s capture-once baseline + "un-minimize target, minimize the rest" makes window i→j reveal j and re-hide i with no explicit teardown — same baseline-replay property that made app re-targeting free in US-010. The controller needed **zero changes** (it already forwards every hover region to the navigator and calls `navigator.close()`→`restore()` on dismiss); the view already rendered window placeholder content (AC4).
- **AC3 "leaving the window ring restores the app's other windows" needs a TARGETED restore that the all-or-nothing `restore()` can't give** (restore() un-hides apps too, which would collapse the app isolation). Added `WindowController.restoreWindows(of: AppID)`: replays just that app's `windowBaseline` (un-minimize + re-raise back-to-front) and **drops only that one baseline key**, leaving `appBaseline` and the session intact. The navigator calls it on every apps-ring (level-0) hover via `restoreWindowIsolation()` BEFORE `expandApp`, so gliding inward from a window to the apps ring (same app → expandApp is a no-op; different app → re-targets) un-minimizes the old windows while the app stays isolated and the sub-wheel stays open. Dropping the baseline key means a later re-isolation of that app re-captures a fresh (restored) baseline — consistent with the capture-once-per-scope model.
- **Restore granularity now has THREE levels**, all on the one capture-once session: `restoreWindows(of:)` (one app's windows, keep apps hidden — leaving the window ring), and `restore()` (everything, end session — dead-zone/close/commit). App re-target (US-010) needs no restore at all (baseline-replay). Match the restore granularity to the gesture: the narrower the hover change, the narrower the restore.
- **Test fixture models the reconciled world:** the `FakeWindowSystem`'s window tokens must equal the enumerator's window numbers (Chrome 11/12, Ghostty 21) so a target carried by a window node resolves to a window the fake actually has — otherwise `hideOtherWindows` minimizes everything (target never matches). This is the same alignment the live `_AXUIElementGetWindow` reconciliation provides in production.
